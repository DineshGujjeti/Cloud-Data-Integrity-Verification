2. System Components

A. Client/User Component

Responsibilities:

  • Upload files to cloud server
  • Split files into fixed-size blocks
  • Build Merkle tree from blocks
  • Store root hash securely locally
  • Generate integrity verification challenges
  • Verify server responses

Data Stored Locally:

  • Root hash of Merkle tree
  • File metadata (name, size, block count)
  • Block hashes
  • Proof paths for verification

Key Operations:

Operation Input Output Complexity
File Split File Blocks O(n)
Tree Building Blocks Merkle Tree O(n log n)
Challenge Gen. File ID, Block Indices Challenge O(k) k=blocks
Response Verify Block Data, Proof Verification Result O(log n)

B. Cloud Storage Server Component

Responsibilities:

  • Receive and store file blocks
  • Maintain file metadata
  • Respond to client challenges
  • Return requested blocks with proofs
  • Maintain operation logs

Data Stored on Server:

  • All file blocks
  • File metadata
  • Client information
  • Operation logs

Server Capabilities:

  • Efficient block retrieval: O(1) per block
  • Challenge response generation: O(k)
  • Scalable storage architecture
  • Redundancy and backup mechanisms

C. Third Party Auditor (TPA) Component

Responsibilities:

  • Independent verification of data integrity
  • Generate random challenges
  • Verify server responses
  • Maintain audit reports
  • Detect tampering or data loss

Audit Process:

  1. Select random blocks to challenge
  2. Send challenge to cloud server
  3. Receive and verify response
  4. Check block integrity
  5. Generate audit report

D. Integrity Verification Module

Key Functions:

  • Merkle tree construction and validation
  • Block hash verification
  • Merkle proof verification
  • Tampering detection
  • Attack simulation for testing

4. Protocol: Merkle Tree Verification

Merkle Proof Verification Algorithm

Input:

  • Block data (B)
  • Block index (i)
  • Proof path (P): List of (direction, hash) pairs
  • Root hash (R): Expected root

Algorithm:

  1. current_hash = SHA256(B)
  2. for each (direction, sibling_hash) in reversed(P):
  3.     if direction == "left":
  4.       current_hash = SHA256(sibling_hash + current_hash)
  5.     else:
  6.       current_hash = SHA256(current_hash + sibling_hash)
  7. return (current_hash == R)

Output:

  • Boolean: True if block is verified, False if tampering detected

Complexity:

  • Time: O(log n) - number of tree levels
  • Space: O(log n) - for proof storage
  • Communication: O(log n) - for transmitting proof

7. Security Analysis

Threat Model

  • Honest-but-Curious Server: Server follows protocol but tries to learn data
  • Malicious Server: Server modifies or deletes blocks
  • Network Attacker: Intercepts and modifies messages
  • Collusion: Server and attacker work together

Defense Mechanisms

  • Merkle Tree: Any single block change is detected
  • SHA-256: Cryptographically secure hashing
  • Random Challenges: Unpredictable audit pattern
  • TPA Verification: Independent audit
  • Root Hash Protection: Secure local storage

Security Guarantees

  • Detects tampering with probability (1 - (1-1/n)^k)
  • Where n = number of blocks, k = challenged blocks
  • After k challenges: 99.5% detection probability for single modified block
